# 外部スキルの取り込み

このガイドを読むと、外部スキルを評価し、導入可否を判断し、正本へ追加して検証・記録するまでの流れが分かります。外部スキルの導入そのものはプロジェクトの判断に委ねますが、判断と手順の正本はこのページです。

## このドキュメントを使う場面

| 場面 | 内容 |
|------|------|
| いつ使うか | 外部リポジトリや共有フォルダから `SKILL.md` を取り込みたいとき |
| 最小トリガー | 「この外部スキルを評価して」とアシスタントに依頼する |
| アシスタントが行うこと | 配布元確認、権限洗い出し、差分提示、dry-run、検証、記録 |
| 確認できる成果物 | 査証ログ、Rulesync 差分、テスト結果 |

## 取り込み前の短いチェック（README 版）

便利さだけで取り込まず、次を確認してから導入してください。

- 配布元、作者、バージョンまたはコミットを特定できるか
- `SKILL.md` と参照するスクリプト・設定・アセットを読めるか
- ファイル書き込み、削除、外部通信、秘密情報、外部 API、課金を要求するか
- 依存パッケージ、対応 OS、必要コマンドがプロジェクトと両立するか
- ライセンスと再配布条件がプロジェクト用途に合うか
- 導入前後の差分、Rulesync dry-run、関連テストを確認できるか

## 1. 取り込む前の評価

- スキルが解決する課題と、既存スキルで代替できない理由を確認します。
- 配布元 URL、作者、取得日、バージョンまたはコミットハッシュ、ライセンスを記録します。
- `SKILL.md` を全文確認し、そこから直接参照されるスクリプト、設定、アセットも確認します。
- トリガー条件が広すぎず、意図しない作業で発火しないことを確認します。
- 外部 API、ネットワーク、認証情報、課金、不可逆操作、プロジェクト外への書き込みを洗い出します。
- Python・Node.js 等の依存追加、OS 固有コマンド、既存ツールとの名前衝突を確認します。

## 2. 取り込み先の判断

- プロジェクトで管理するスキルは `.rulesync/skills/<skill-name>/` を正本候補とします。
- Rulesync 生成物（`.agents/`、`.claude/`、`.cursor/` 等）だけを直接編集して正本扱いしません。
- 個人環境だけで使うスキルと、プロジェクトへ同梱するスキルを分けます。
- 外部スキルをそのまま再配布できるか、ライセンスを確認します。

## 3. 導入手順

1. 既存ファイルと導入予定ファイルを確認します。
2. 変更予定、依存追加、権限・外部通信の有無を提示します。
3. 必要な承認を得てから正本へ追加します。
4. Rulesync の dry-run で生成先と差分を確認します。

```bash
corepack pnpm dlx rulesync generate --dry-run
```

5. 問題がなければ本番生成し、Git 差分と生成されたスキルを確認します。

```bash
corepack pnpm dlx rulesync generate
```

## 4. 検証と記録

- スキルの代表的なトリガー文で期待する動作になるか確認します。
- 誤発火しやすい近接タスクでも意図せず起動しないか確認します。
- 付属コードがある場合は対応テストを実行します。
- 導入元、版、ライセンス、変更点、検証結果を査証ログへ記録します。

```bash
python tools/kernel/workspace_audit_log.py append "外部スキル foo を導入。配布元: ... 検証: dry-run OK, tests OK"
```

更新時も初回導入と同じ確認を行い、無条件に最新版へ置き換えません。

## 受入目安

| 判定 | 目安 |
|------|------|
| 取り込み可 | 配布元・版・ライセンスが明確で、動作と権限を確認でき、差分とテストで検証可能 |
| 条件付き | 有用だが外部通信、API キー、課金、広い書き込み権限があるため、用途限定と明示承認が必要 |
| 取り込み見送り | 配布元不明、難読化、参照コードを確認できない、不要な秘密情報要求、ライセンス不明、不可逆操作を隠す |
